Java序列化机制

前言

通常在进行网络传输时，序列化数据使用的都是XML，或JSON格式，但是这两种格式，均无法传输对象类型的数据，因此Java提供了特有的序列化机制，可以用来传输对象类型的数据。

Java序列化(writeObject)

序列化是指把一个Java对象变成二进制内容，本质上就是一个byte[]数组。

byte[]数组可以通过网络传输，或者写入文件中。

Java对象可以进行序列化的前提是，对象所属的类 必须 实现一个特殊java.io.Serializable接口，且该类的所有 成员变量 必须是可序列化的，如果有一个 成员变量 不是可序列化的，则该 成员变量 必须注明是transient短暂的。

Serializable接口没有定义任何方法，它是一个空接口，可以理解为是一个"标记接口"，给JVM看的。

序列化：Java通过java.io.ObjectOutputStream类中的writeObject(Object obj)方法对参数指定的obj对象进行序列化，把字节序列写到一个目标输出流中，按Java的标准约定是给文件一个.ser扩展名。

Java反序列化(readObject)

反序列化: Java通过java.io.ObjectInputStream类中的readObject()方法 从一个源输入流中读取字节序列，再把读取到的字节序列 反序列化为一个对象，并将对象返回。

Test.java

Java反序列化时，由JVM直接构造出Java对象，不调用构造方法，构造方法内的代码，在反序列化时不会被执行。

readObject()可能抛出的异常有:

  • ClassNotFoundException：没有找到对应的Class；
  • InvalidClassException：Class不匹配。

对于ClassNotFoundException，这种情况常见于一台电脑A上的Java程序把一个Java对象，例如，Student对象序列化以后，通过网络传给另一台电脑B上的另一个Java程序，但是电脑B的Java程序并没有定义Student类，所以无法反序列化。

对于InvalidClassException，这种情况常见于
序列化的Student对象定义的成员变量age为int类型，但是反序列化时，
Student类定义的age字段类型 被改成了long类型，所以导致class不兼容。

为了避免这种Class定义变动导致的不兼容，Java序列化允许Class定义一个特殊的serialVersionUID静态变量，用于标识Java类的序列化”版本”，通常可以由IDE自动生成。

如果增加或修改了字段，可以改变serialVersionUID的值，这样就能自动阻止不匹配的Class版本。

小结

  • Java的序列化机制仅适用于Java。
  • Java序列化后的数据本质上是一个byte[]数组。
  • 只有实现了Serializable 接口的类才能被序列化。

  • 不想将 类中的 所有 成员变量 都进行序列化，可以对成员变量声明 transient。

    private transient String name;

    当对该类序列化时，会自动忽略被 transient 修饰的成员变量。

  • 反序列化时不会调用构造方法。
  • 设置serialVersionUID作为版本号（非必需），用以避免Class定义变动导致的不兼容。

RMI（Java Remote Method Invocation）

RMI（Remote Method Invocation）为远程方法调用，是允许运行在一个Java虚拟机的对象调用运行在另一个Java虚拟机上的对象的方法。

这两个虚拟机可以是运行在 相同计算机上 的不同进程中，也可以是运行在网络上的不同计算机中。

一个完整的RMI过程，由客户端与服务端组成。

一个RMI Server分为三部分：

1、一个继承了java.rmi.Remote类的接口（这里是IHey），接口内包含了远程调用的方法声明。

2、一个继承java.rmi.server.UnicastRemoteObject(也可以不继承) 并 实现上述接口中所有方法体的类（这里是HeyImpl）

注意：我们必须显式地提供构造函数，即使构造函数不执行任何操作，因为构造函数（像任何方法一样）也可以引发RemoteException异常。（我们不能使用自动生成的默认构造函数，因为它不会引发异常）
